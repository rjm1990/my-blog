代码审查是提升代码质量和团队能力的最佳实践。

## 🎯 代码审查的价值

- 发现潜在bug
- 分享知识
- 保持代码风格一致
- 帮助新人成长
- 减少单点依赖

## ✅ 如何做Code Review

### 审查什么？

- **逻辑正确性**: 代码是否实现了需求
- **代码质量**: 可读性、可维护性
- **性能**: 是否有明显性能问题
- **安全**: 是否有安全隐患
- **测试**: 是否有足够的测试

### 审查技巧

1. **先理解需求**: 不知道要解决什么问题，就无法判断方案是否合理
2. **先看整体**: 先看架构设计，再看细节
3. **关注重点**: 不要纠结于代码格式（用工具自动处理）
4. **提建设性意见**: 不要只说"不好"，要说"怎么改"

### 评论的艺术

```markdown
<!-- ❌ 不好的评论 -->
这代码写得不好。

<!-- ✅ 好的评论 -->
这里用循环会有性能问题，建议用Map优化查询：
\`\`\`javascript
const userMap = new Map(users.map(u => [u.id, u]));
const user = userMap.get(userId);
\`\`\`
```

## 🤔 如何被Review

### 提交前自检

- [ ] 代码能正常运行
- [ ] 写了单元测试
- [ ] 没有无用的代码/注释
- [ ] 遵循团队代码规范
- [ ] commit message清晰

### 减少Review难度

- **控制PR大小**: 一个PR不超过400行
- **提供上下文**: PR描述说明改了什么、为什么改
- **添加截图**: UI改动附上截图
- **自己先Review一遍**: 提交前自己先看一遍diff

### 面对评论

- **不要情绪化**: 代码审查不是人身攻击
- **虚心接受**: 别人的建议可能有道理
- **主动讨论**: 不理解的地方要问
- **及时响应**: 不要让PR挂着不动

## 📋 Code Review Checklist

### 提交者自查清单

```markdown
## 功能
- [ ] 实现了需求描述的功能
- [ ] 边界情况已处理
- [ ] 错误处理完善

## 代码质量
- [ ] 代码可读性好
- [ ] 没有重复代码
- [ ] 变量/函数命名清晰

## 测试
- [ ] 单元测试覆盖
- [ ] 测试用例有意义

## 文档
- [ ] 接口文档已更新
- [ ] README已更新（如有必要）
```

### 审查者检查清单

```markdown
## 设计
- [ ] 架构设计合理
- [ ] 不过度设计

## 实现
- [ ] 逻辑正确
- [ ] 性能可接受
- [ ] 安全无隐患

## 可维护性
- [ ] 代码易懂
- [ ] 易于扩展
```

## 🚫 常见反模式

1. **LGTM滥用**: 不仔细看就通过
2. **挑刺过度**: 纠结于个人偏好
3. **积压PR**: PR挂了几周不看
4. **大型PR**: 一次改几千行

## 💡 团队实践建议

- **设定SLA**: PR应在24小时内得到响应
- **自动化检查**: 用CI自动检查代码风格、测试
- **定期分享**: 分享代码审查中发现的问题
- **配对编程**: 复杂功能用配对编程替代审查
