面试是双向选择，不只是公司在选你，你也在选公司。

## 📋 面试准备

### 简历准备

**简历结构**
```markdown
# 基本信息
姓名 | 电话 | 邮箱 | GitHub

# 教育背景
学校 | 专业 | 时间

# 工作经历
公司 | 职位 | 时间
- 负责XX项目，使用XX技术
- 优化了XX性能，提升了XX%
- 解决了XX问题，影响XX用户

# 项目经历
项目名称 | 角色 | 时间
- 项目背景：解决了什么问题
- 技术方案：用了什么技术，为什么
- 个人贡献：做了什么
- 项目成果：量化结果

# 技术栈
语言 | 框架 | 工具
```

**简历要点**
- 量化成果：提升了50%，而不是"性能优化"
- 突出重点：最近的工作最重要
- 针对性：针对岗位调整简历
- 简洁：1-2页即可

### 技术准备

**必准备内容**
- [ ] 数据结构：数组、链表、树、图
- [ ] 算法：排序、查找、动态规划
- [ ] 计算机基础：网络、操作系统
- [ ] 语言深入：常用语言的原理
- [ ] 项目经验：项目的细节、难点

**刷题策略**
- LeetCode Hot 100
- 每道题要理解原理
- 能举一反三
- 自己写，不要只看答案

### 行为面试准备

**常见问题**
- 你最大的成就是什么？
- 遇到的最大挑战是什么？
- 你的优缺点？
- 为什么离职？
- 为什么选择我们？

**回答框架**
```
情境(S) → 任务(T) → 行动(A) → 结果(R)
```

## 🎯 面试类型

### 电话面试（30-60分钟）

**特点**
- 快速筛选
- 基础问题为主
- 可能在线coding

**准备**
- 安静环境
- 准备纸笔
- 电脑在手边
- 保持清醒

### 技术面试（1-2小时）

**内容**
- 算法题
- 系统设计
- 项目深挖
- 技术原理

**技巧**
- 先理解题目，再开始写
- 说出你的思路
- 先写brute force，再优化
- 注意边界条件

### 系统设计（45-60分钟）

**框架**
```
1. 明确需求
   - 功能需求
   - 非功能需求（性能、可用性）
   
2. 容量估算
   - 用户量、QPS
   - 存储、带宽

3. 系统架构
   - 整体设计
   - 数据库设计
   - API设计

4. 深入细节
   - 核心功能实现
   - 性能优化
   - 可扩展性
```

### 行为面试（30-60分钟）

**考察点**
- 沟通能力
- 团队协作
- 解决问题能力
- 文化匹配

**准备**
- 准备3-5个故事
- 用STAR法则组织
- 诚实回答

## 💡 面试技巧

### 编码题

1. **理解题目**
   - 重复题目
   - 确认输入输出
   - 问清楚边界条件

2. **沟通思路**
   - 先说思路，再写代码
   - 解释为什么这么做
   - 讨论时间和空间复杂度

3. **写代码**
   - 先写主体逻辑
   - 变量命名清晰
   - 边界处理

4. **测试**
   - 用例子测试
   - 考虑边界情况
   - 修复bug

### 项目深挖

**可能的问题**
- 为什么这么设计？
- 有什么问题？怎么解决的？
- 如果重新设计，会怎么做？

**准备要点**
- 了解项目的每个细节
- 知道技术选型的原因
- 总结项目的得失

### 提问环节

**好问题**
- 团队的技术栈是什么？
- 团队面临的挑战是什么？
- 公司的技术文化是怎样的？
- 这个岗位的日常工作是什么？

**不要问**
- 薪资（HR会谈）
- 太基础的信息（自己查）

## 🚫 面试误区

1. **只刷题不思考**: 要理解原理
2. **背答案**: 面试官能看出来
3. **不沟通**: 默默写代码是大忌
4. **放弃太快**: 不会的题也要尝试

## 📊 面试评估维度

```
技术能力 40%
- 算法能力
- 系统设计
- 项目经验

软技能 30%
- 沟通能力
- 团队协作
- 学习能力

文化匹配 30%
- 价值观
- 工作方式
- 发展潜力
```

## 💰 谈薪技巧

**时机**
- 拿到Offer后再谈
- 了解市场行情

**策略**
- 了解薪资结构
- 不要第一个说数字
- 可以谈判
- 看总包，不只看base

## 📝 面试复盘

每次面试后：
1. 记录问题
2. 总结不会的知识点
3. 补充学习
4. 下次改进
